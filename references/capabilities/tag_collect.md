# 标签选品采集指南（tag_collect）

> 命令为只读/导出型；Web 工作台默认真实数据模式，通过 Playwright/RPA 打开 1688 页面采集和详情核验。开发样例模式必须显式开启，不能作为正式测试结果。

## CLI 调用

启动本地 Web 筛选测试工作台：

```bash
python3 {baseDir}/cli.py tag_collect --serve --port 8765
```

命令行采集并导出：

```bash
python3 {baseDir}/cli.py tag_collect --categories "家居日用品,收纳清洁" --tags "微信小店,一件代发,48小时发货" --sample-data
```

| 参数 | 默认 | 说明 |
|------|------|------|
| `--serve` | false | 启动本地 Web 筛选测试工作台 |
| `--host` | `127.0.0.1` | Web 监听地址 |
| `--port` | `8765` | Web 监听端口 |
| `--categories` | `""` | 复选类目标签，逗号分隔 |
| `--tags` | `""` | 复选运营标签，逗号分隔 |
| `--keywords` | `""` | 额外搜索词，逗号分隔；不传则由类目和搜索型标签生成 |
| `--source-urls` | `""` | 直接采集真实 1688 搜索页或商品详情页 URL；适合账号登录受阻时测试公开页面真实数据 |
| `--exclude-tags` | `""` | 排除标签；命中标题、类目、风险提示或标签来源时过滤，并写入任务快照 |
| `--max-queries` | `20` | 最大查询词数量 |
| `--max-items-per-query` | `20` | 每个查询词候选扫描预算上限 |
| `--target-publishable-count` | `0` | 目标可铺商品数量；不传则沿用候选预算 |
| `--output-format` | `xlsx` | 导出格式：`xlsx` 或 `csv` |
| `--collect-source` | `rpa` | 真实采集来源：`rpa` 打开真实 1688 页面；`api` 调用 AK 接口 |
| `--sample-data` | false | 开发样例模式，不调用 1688；正式测试不要开启 |

## Web 工作台使用

- 推荐入口：`python3 cli.py tag_collect --serve --port 8765`，再打开 `http://127.0.0.1:8765`。
- 页面默认关闭“开发样例”，即默认采集真实 1688 页面数据。若弹出登录或验证页面，用户需要在浏览器中扫码/处理验证。
- 标签不会全部拼成搜索词。`一件代发`、`48小时发货`、`7天包退货/七天无理由`、`批发包邮`、`实力商家` 等会进入 `native_filters`，由 RPA 在 1688 搜索页尝试点击/勾选。
- 截图框出的搜索页服务筛选已接入 `native_filters`：`严选`、`分销严选`、`一件代发`、`退货包运费`、`7天无理由退货`、`24H发货`、`48H发货`、`官方物流`、`密文面单`、`晚揽必赔`、`24H支揽率`、`48H支揽率`、`合并供应商`。同义筛选会按 key 合并别名并保留执行记录。
- `所在地`、`商家特色`、`经营模式` 属于下拉筛选，RPA 会先尝试展开筛选组再点击选项；执行结果仍以 `clicked/not_found/click_failed` 为准。`店铺商品数` 当前为预留字段，不会伪造成已过滤结果。
- 截图中的 `商品星级`、`好评率`、`评价数`、`评价标签` 属于详情评价块字段。系统会先保留候选，进入详情页核验后再重评估筛选；新增导出字段为 `5.10 商品星级`、`5.11 评价标签`。
- 详情核验后未满足星级、好评率、评价数或评价标签筛选的商品，会从主结果移入 `filter_excluded_rows`，并出现在导出的 `异常复核` sheet 中供审计。
- 如果当前类目或页面没有某个原生筛选项，系统会记录 `not_found` 并在 Web、JSON 快照、XLSX `筛选执行记录` 中提示，不能把它当作已筛选成功。
- 好评率、品退率、发货率、评论数、复购率、近30天销量、代发订单量支持区间标签；页面读不到的字段保持待详情页核验或人工复核，不猜值。
- 类目区支持一级/二级/三级路径。当前 `category_dict.json` 是本地种子字典，状态为 `partial_seed_needs_official_sync`，后续仍需采集/确认完整官方 1688 类目字典。
- 如果账号登录不上或一直触发校验，可以在“1688 页面 URL”粘贴浏览器里能打开的搜索页/商品详情页链接。URL 模式仍然读取真实页面，不会使用样例数据；但公开页面看不到的运费、品退率、发货率等字段仍会保持待核验或失败。
- 如果出现“访问被拒绝”或滑块反复失败，系统会返回 `security_verification_required` 并暂停任务：不会继续刷新、不会导出、不会自动重试。此时应由人工正常浏览 1688，完成登录/验证后，再把已能打开的搜索页/详情页 URL 粘贴回来继续。
- 风控较敏感时，推荐采用“人工浏览 + 系统读取已加载页面/候选 URL”的低风险模式：先由人工在 1688 页面勾选原生筛选并浏览候选商品，再把候选 URL 或已加载页面交给系统做去重、评分、详情核验和导出。不要使用代理池、验证码绕过、反爬逆向或高频自动重试。
- RPA 默认使用 `TAG_COLLECT_RPA_PACING=human` 慢速节奏：页面加载、筛选点击、滚动、详情核验都会加入随机等待和分段滚动。Web 默认每词采集 5 条、详情核验 5 条；真实账号测试不要一开始拉高数量。仅开发排查时可临时设为 `TAG_COLLECT_RPA_PACING=fast`。
- Web 工作台默认开启“详情补字段/自动核验”，采集列表后会按上限将高潜商品进入详情页补充运费、品退率、发货率等关键字段；默认上限 3 个，遇到登录/滑块/访问被拒绝会立即停止后续详情页访问。关闭该开关时，采集只做列表初筛，可再手动点击“真实详情核验”。
- Web 的“目标可铺数”对应 `target_publishable_count`，按系统规则预判的 `可铺` 数量计入达标；`谨慎` 只作为可观察候选，不计入满足投放数量。
- 系统会记录本地历史筛除池 `{TAG_COLLECT_DATA_DIR}/rejected_products.json`。同一筛选签名下，列表硬筛失败、排除标签命中、详情后筛选失败或详情后明确“不建议”的商品，下次采集会先跳过，减少重复详情核验和重复人工判断。
- 详情核验后会重新计算推荐分、推荐等级和微信小店建议，并写入 `rescore_records`；导出 `标签配置` sheet 会显示目标可铺数、候选扫描、历史跳过和短缺原因。
- 采集结果会返回 `automation_state`，用于区分接口成功和业务流程状态：
  - `pending_detail`：列表初筛完成，但仍有高潜商品待详情页核验。
  - `paused`：列表采集成功，但详情页遇到登录、滑块、验证码或访问拒绝，已停止后续详情核验，需要人工接管。
  - `partial` / `failed`：详情字段未完整可信，导出后重点复核异常/边界商品。
  - `verified` / `sample_verified`：详情字段已完成本轮核验；其中 `sample_verified` 只代表样例流程通过。
- `.xlsx` 的 `标签配置` sheet 会写入任务状态、当前阶段、建议动作、详情核验暂停原因、已核验/失败/待核验数量，避免把“列表成功”误读成“详情字段都可信”。
- 样例详情核验会把行状态更新为 `sample_verified`，并生成字段级核验记录。`sample_verified` 只代表流程和字段映射已通过样例验证，不代表真实 1688 数据可信。
- `partial_verified` 表示真实详情页只提取到部分字段；未提取到的运费、品退率、发货率等字段必须人工复核。
- 如果通过 `--host 0.0.0.0` 让其他设备访问页面，真实采集仍会被服务端禁用并锁定为远程只读/样例模式；当前不是多人真实采集服务。
- Web 第一版不是绕过 1688 风控的爬虫，不做验证码绕过、反爬规避或高频页面抓取。

## 输出结构

Agent 收到的 CLI 标准输出：

```json
{
  "success": true,
  "markdown": "## 标签选品采集结果\n\n- 采集批次：`20260613_120000_000`\n- 查询词：家居日用品 防晒\n- 商品数：3\n- 导出文件：`.../tag_collect_20260613_120000_000.xlsx`\n...",
  "data": {
    "run_id": "20260613_120000_000",
    "queries": ["家居日用品 防晒"],
    "row_count": 3,
    "output_path": ".../tag_collect_20260613_120000_000.xlsx",
    "snapshot_path": ".../tag_collect_20260613_120000_000.json",
    "sample_data": false,
    "columns": ["序号", "商品类目", "商品主图"],
    "filter_plan": {"native_filters": [], "post_filters": []},
    "filter_results": [],
    "top_items": [],
    "verification_queue": [],

```

展示时先输出 `markdown` 字段，再补充面向用户的下一步建议。不要把导出表当作已核验铺货清单。

## 字段编号

字段编号采用“分组编号.组内序号”：

- `1.1`：第 1 组的第 1 个字段，如“序号”
- `1.2`：第 1 组的第 2 个字段，如“商品类目”

该编号用于字段确认和验收沟通，不是小数、优先级或版本号。用户说“保留 1.1/1.2、调整 2.4”时，Agent 应按字段编号理解并追踪到对应导出列。

## 人工复核规则

导出后仍保留人工复核列，但复核重点是 `谨慎`、字段缺失、风控暂停、品牌/IP 不确定和规则冲突商品；系统已明确不满足筛选的商品会进入历史筛除池。Agent 应提醒用户在表格中更新：

- `人工复核状态`
- `人工复核备注`
- `微信小店铺货建议`

列表字段只做初筛。运费、是否包邮、是否一件代发、品退率、发货率、24 小时揽收率、发货时效、起批量、SKU、供应商可靠性等关键字段，仍需进入详情页或可信来源核验后才能作为最终铺货依据。

## 导出工作表

`.xlsx` 当前包含：

- `选品结果`
- `字段说明`
- `标签配置`
- `异常复核`
- `核验记录`
- `筛选执行记录`

`核验记录` 保存字段级记录：商品 ID、字段键、字段名、原始值、标准化值、来源、来源链接、模式、状态、核验时间、失败原因。
`筛选执行记录` 保存 1688 原生筛选动作：筛选键、标签、显示名称、状态、来源、查询/页面、页面 URL、匹配文本、提示。

## 异常处理

| 场景 | 表现 | Agent 应对 |
|------|------|-----------|
| Web 端口被占用 | 启动失败或无法监听 | 换一个端口，例如 `--port 8766` |
| 未安装 Playwright | `未安装 playwright` | 执行 `npm install` 和 `npm run install-browsers` |
| RPA 弹出登录页 | 浏览器停在登录/验证页或返回 `login_required` | 用户扫码登录或处理验证；如果账号登录不上，先粘贴浏览器里能打开的 1688 URL 测试公开页面真实数据 |
| 一直触发安全校验/滑块验证或访问被拒绝 | 返回 `security_verification_required`、`action=manual_handoff`、`retryable=false`，不会绕过验证码，也不会继续采集或导出 | 停止反复刷新；用户在真实浏览器中手动完成验证，或切换为人工浏览后粘贴已打开的搜索页/商品详情页 URL；也可使用已登录且已通过验证的 Chrome CDP 会话 |
| 列表成功但详情页触发风控 | `automation_state.status=paused`，`verification_stopped_reason` 有值，Excel `标签配置` 写入暂停原因 | 说明“列表初筛成功，但详情字段未完整可信”；不要直接铺货，等待人工处理验证后再继续详情核验 |
| API 真实采集无 AK | `AK 未配置` | 引导用户先 `configure`，或改用默认 RPA 真实页面采集 |
| 导出后字段未核验 | `verification_status: unverified` 或字段为“待详情页核验” | 明确这是初筛结果，不能直接进入正式铺货结论 |
| 原生筛选项找不到 | `filter_results.status: not_found` | 告知用户当前类目/页面没有展示该筛选项，不能当成筛选成功；必要时换类目或手动打开 1688 页面确认 |
| 样例详情核验完成 | `verification_status: sample_verified` | 明确这是样例核验结果，不是真实 1688 详情页核验 |
| 真实采集限流/授权异常 | HTTP 401/429 等 | 按 `references/common/error-handling.md` 处理 |
