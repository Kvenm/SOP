# SOP 项目中文说明

## 1. 项目定位

SOP 是一个面向 1688 选品运营的本地工具项目。当前第一版重点解决：

- 按一级/二级类目和运营标签进行复选筛选。
- 根据标签组合生成搜索词和过滤规则。
- 采集候选商品并进行列表字段初筛、去重、评分和风险标记。
- 对高潜商品形成详情页核验队列。
- 补充样例详情核验记录，验证运费、品退率、发货率等关键字段的导出链路。
- 导出 Excel/CSV，导出后再由运营人工复核。

当前项目不是绕过 1688 风控的爬虫，也不是正式的多人 SaaS。第一版目标是把基线流程跑通，便于后续接入真实 1688 AK、商品详情接口或低频 RPA 核验。

## 2. 当前已实现功能

### 2.1 标签选品 Web 工作台

启动命令：

```bash
python3 cli.py tag_collect --serve --port 8765
```

浏览器打开：

```text
http://127.0.0.1:8765
```

已支持：

- 一级类目复选。
- 二级类目复选。
- 运营筛选标签复选。
- 搜索词、排除标签、查询词上限、每词商品上限配置。
- 样例数据模式，默认开启，不需要 AK，不登录 1688。
- 结果表格筛选：商品/类目、最低分、推荐等级、核验状态、铺货建议。
- 字段编号页，字段按 `1.1`、`1.2`、`2.1` 这种方式编号。
- 下载 XLSX 或 CSV。
- “样例核验高潜”按钮，用于验证 P0/P1 高潜商品的字段核验流程。

### 2.2 导出文件

XLSX 当前包含 5 个工作表：

- `选品结果`
- `字段说明`
- `标签配置`
- `核验失败`
- `核验记录`

导出流程符合当前需求顺序：

```text
标签模板配置
-> 创建采集任务
-> 标签组合生成搜索词和过滤规则
-> 调用 1688 搜索或样例采集候选商品
-> 列表字段初筛
-> 商品去重
-> 高潜商品进入详情页核验队列
-> 补充运费、品退率、发货率等关键字段
-> 综合评分和风险标记
-> 导出 Excel 或 CSV
-> 人工复核
-> 可选：进入铺货 dry-run 流程
```

注意：人工复核在导出之后进行。表格中已预留 `人工复核状态`、`人工复核备注`、`微信小店铺货建议(人工复核)`。

### 2.3 字段真实性

当前用字段状态区分数据可信度：

- `unverified`：未进入详情页核验，只能作为初筛数据。
- `sample_verified`：样例详情核验通过，只代表流程可用，不代表真实 1688 数据可信。
- `verified`：预留给后续真实详情页核验。
- `failed`：详情核验失败或字段缺失。

关键字段在详情核验前不会伪装成可信值，会显示为 `待详情页核验`。重点字段包括：

- 批发运费
- 代发运费
- 是否包邮
- 品退率
- 24 小时揽收率
- 发货率
- 发货时效
- 是否一件代发
- 起批范围
- 月代发订单
- 店铺名称
- 公司类型
- 是否源头工厂
- 库存

## 3. 当前未完成和边界

以下功能尚未完成，不能在测试时误认为已经可用：

- 真实 1688 商品详情页字段解析尚未接入。
- Playwright/RPA 扫码登录核验尚未接入。
- 在线文档同步尚未接入，目前是 XLSX/CSV 导出。
- 官方完整 1688 类目树尚未自动同步，目前是第一版类目池和二级类目镜像。
- 多人账号、权限、任务队列、审计日志尚未产品化。
- 自动铺货只建议先做 dry-run，不建议直接批量写入。

## 4. 安装与本地运行

### 4.1 环境要求

- Python 3
- 本地浏览器
- 如需真实 1688 搜索，需要配置 1688 AK

### 4.2 克隆项目

```bash
git clone <你的 GitHub SOP 仓库地址>
cd SOP
```

### 4.3 启动样例工作台

```bash
python3 cli.py tag_collect --serve --port 8765
```

默认监听本机地址 `127.0.0.1`。如果只是同一台电脑测试，使用默认即可。

### 4.4 在其他设备访问本机工作台

如果要让同一局域网内其他设备访问，可以启动：

```bash
python3 cli.py tag_collect --serve --host 0.0.0.0 --port 8765
```

然后在另一台设备访问：

```text
http://<运行电脑的局域网 IP>:8765
```

注意：

- 该方式只建议用于样例数据测试。
- 不要在公网暴露该端口。
- `--allow-real` 真实采集只允许本机地址开启，避免把 AK 能力暴露给局域网或公网。

## 5. 样例采集测试步骤

1. 启动工作台。
2. 保持“样例数据”勾选。
3. 选择一级类目，例如 `女装/女士精品` 或 `家用电器`。
4. 选择运营标签，例如 `微信小店`、`一件代发`、`48小时发货`。
5. 点击“运行采集”。
6. 在结果页确认有商品、推荐分、风险提示和待核验高潜队列。
7. 点击“样例核验高潜”。
8. 确认批发运费、品退率、发货率等字段从 `待详情页核验` 变为样例核验值。
9. 下载 XLSX，检查 5 个工作表和字段说明。

也可以直接跑命令行样例采集：

```bash
python3 cli.py tag_collect --sample-data --categories "女装/女士精品" --tags "微信小店,一件代发,48小时发货" --output-format xlsx
```

## 6. 真实采集与 AK

AK 是访问 1688 平台接口的身份凭证。未配置 AK 时，真实采集会被阻止。

检查配置：

```bash
python3 cli.py check
```

配置 AK：

```bash
python3 cli.py configure YOUR_AK
```

启动允许真实搜索的本地工作台：

```bash
python3 cli.py tag_collect --serve --port 8765 --allow-real
```

真实模式说明：

- 真实候选商品采集复用现有 `search` capability。
- 真实详情核验尚未完成，运费、品退率、发货率等字段仍需后续接入 `prod_detail` 或 RPA。
- 当前不要把真实采集结果直接当作最终铺货依据。

## 7. 风控和封号风险

采集 1688 时存在平台风控风险，尤其是高频访问、模拟登录、验证码绕过、异常 IP、批量详情页打开等行为。

当前第一版的风控策略：

- 默认使用样例数据，不调用 1688。
- 真实采集必须显式加 `--allow-real`。
- Web 真实采集仅允许监听本机地址。
- 服务端限制查询词数量和每词商品数量。
- RPA 定位为低频详情页核验，不作为主采集引擎。

后续真实 RPA 接入时建议：

- 使用真实账号扫码登录，不保存账号密码。
- 控制频率，优先只核验 P0/P1 高潜商品。
- 遇到验证码或风控提示时暂停，不绕过平台限制。
- 所有真实字段保留来源、时间、状态和失败原因。

## 8. 多 agent 流程

项目保留了可复用多 agent workflow：

```text
.codex-workflows/workflows/tag-collect-next.workflow.js
```

该流程包含：

- 产品 agent：确认需求、验收标准和不该提前做的功能。
- 架构 agent：评审 capability、数据结构、导出和详情核验架构。
- 验证/风控 agent：补测试用例、风险点和用户可见验收点。

运行记录目录 `.codex-workflows/runs/` 不提交到 GitHub，只保留 workflow 模板。

## 9. 发布到 GitHub 前检查

提交前建议执行：

```bash
env PYTHONPYCACHEPREFIX=/private/tmp/pycache-1688-shopkeeper \
  python3 -m py_compile scripts/capabilities/tag_collect/service.py \
  scripts/capabilities/tag_collect/web.py \
  scripts/capabilities/tag_collect/cmd.py cli.py

python3 scripts/capabilities/tag_collect/smoke_test.py
```

不要提交：

- AK 或任何真实密钥。
- `.env`、`.env.local`、`.env.*`。
- `.local-data/`。
- `1688-skill-data/`。
- `.codex-workflows/runs/`。
- `output/` 截图和本地验收产物。
- XLSX、CSV、日志、采集快照 JSON。

## 10. 推荐下一阶段

下一阶段优先级：

1. 接入 `prod_detail` 真实详情快照，做字段白名单映射。
2. 增加 `partial_verified` 状态，区分部分字段核验成功。
3. 接入 Playwright/RPA 扫码登录，仅低频核验 P0/P1 高潜商品。
4. 引入真实 1688 类目同步或类目导入机制。
5. 增加在线文档同步。
6. 再考虑多人化：用户、任务、权限、队列和审计。
