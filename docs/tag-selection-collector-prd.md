# 标签选品自动化采集工具 PRD

## 1. 背景与目标

当前需求不是单次搜索商品，而是建设一个面向 1688 选品运营的自动化采集工具：

1. 运营先配置可复选的标签与筛选条件。
2. 系统根据标签组合在 1688 上自动采集候选商品。
3. 候选商品进入初筛、去重、评分。
4. 关键字段不能只相信列表页或插件抓取结果，必须进入商品详情页或可信来源页二次核实。
5. 最终产出可人工复核的 Excel 或在线文档，作为后续铺货、对比、运营跟进的基础表。

核心目标是形成一条可审计、可复用、可扩展的选品采集流程，而不是只做一个临时爬取脚本。

## 2. 用户与场景

### 2.1 目标用户

- 1688 选品运营人员
- 店铺运营人员
- 负责抖店、拼多多、小红书、淘宝铺货的业务人员
- 后续可服务多 agent 自动选品流程中的选品 agent、风控 agent、导出 agent

### 2.2 核心场景

1. 运营按品类、平台、价格带、履约、竞争、风险等标签批量找货。
2. 运营希望快速筛出一批可进一步验证的商品，而不是手动逐个搜索。
3. 运营不信任列表页或插件直接摘取的数据，希望对运费、品退率、发货率等关键字段做详情页核验。
4. 运营需要把结果导出到 Excel 或在线文档，方便评审、分工、复核和后续铺货。

## 3. 基线原则

### 3.1 标签必须可复选

标签不是单选分类，而是多条件组合。一个采集任务可以同时包含：

- 多个品类标签
- 多个平台标签
- 多个价格带标签
- 多个履约标签
- 多个销售表现标签
- 多个风险控制标签

系统需要支持标签之间的关系：

- AND：必须同时满足，例如“抖店 + 50 元内 + 高发货率”
- OR：满足任一即可，例如“收纳 OR 清洁”
- EXCLUDE：排除，例如“排除品牌风险”

### 3.2 列表页数据只做初筛

列表页、搜索接口或插件摘取的数据可用于快速初筛，但不能作为最终可信结果。

需要区分字段可信等级：

| 等级 | 含义 | 使用方式 |
| --- | --- | --- |
| 初筛字段 | 来自搜索结果或列表页 | 可用于排序、初筛、候选池生成 |
| 待核验字段 | 来源可能不稳定或容易误读 | 必须进入详情页核验后才能导出为可信字段 |
| 核验字段 | 已从商品详情页、商家页或可信接口确认 | 可作为最终导出字段 |

### 3.3 关键经营字段必须二次核实

以下字段在导出前必须有核验状态：

- 运费
- 是否包邮
- 是否一件代发
- 品退率
- 发货率
- 24 小时揽收率
- 发货时效
- 起批量
- SKU/规格
- 商家类型
- 店铺/供应商可靠性

### 3.4 导出结果必须保留数据来源

每个导出字段需要尽量保留：

- 来源页面或接口
- 采集时间
- 是否已核验
- 核验失败原因
- 原始值
- 标准化后的值

## 4. 总体流程

```text
标签模板配置
-> 创建采集任务
-> 标签组合生成搜索词和过滤规则
-> 调用 1688 搜索或采集候选商品
-> 列表字段初筛
-> 商品去重
-> 高潜商品进入详情页核验
-> 补充运费、品退率、发货率等关键字段
-> 综合评分和风险标记
-> 导出 Excel 或同步在线文档
-> 人工复核
-> 标记是否适合微信小店/其他下游平台铺货
-> 可选：进入铺货 dry-run 流程
```

人工复核必须发生在导出之后。系统先产出标准化选品表，运营再基于导出表做人工确认、补充备注、筛掉不适合微信小店或其它下游平台的商品。

### 4.1 当前 Web 筛选测试工作台

当前第一版已提供本地 Web 筛选测试工作台，启动命令：

```bash
python3 cli.py tag_collect --serve --port 8765
```

工作台定位是验证标签复选、字段编号、结果表格、导出文件和人工复核链路。页面默认使用样例数据模式，不调用 1688 接口，适合做筛选规则与导出验收。若关闭样例数据并发起真实采集，必须先配置 AK，由现有 `search` 能力调用 1688 数据；没有 AK 时工作台应阻止真实采集并提示切回样例数据或先配置 AK。

Web 第一版不是绕过 1688 风控的爬虫，不承担登录态模拟、验证码绕过、页面反爬规避或高频抓取。涉及真实平台数据时必须使用项目已有 AK 与接口能力，并保留限流、授权、字段核验和人工复核边界。

## 5. 标签体系

标签体系分为两层：

1. 1688 平台类目标签：直接参照 1688 平台类目树，用于决定采集范围。
2. 运营筛选标签：在类目采集结果上继续筛选、评分和复核。

品类标签不再由项目手工维护少量示例，而是以 1688 平台真实类目为基准。采集到商品后，必须保留 1688 原始类目路径，例如 `居家日用品>挡风、遮阳、防雨工具>连体雨衣、雨披`。

## 6. 筛选字段体系（确认编号版）

## 7. 评分与筛选逻辑

MVP 采用规则评分，不依赖黑盒模型直接给结论。

### 7.1 基础评分方向

- 销量表现：近 30 天销量、累计销量、代发销量
- 竞争强度：下游铺货数、同款风险
- 质量口碑：好评率、评价数、复购率、品退率
- 履约能力：24 小时揽收率、发货率、发货时效
- 成本空间：价格、运费、包邮、起批量
- 平台适配：目标平台和商品属性是否匹配
- 风险扣分：品牌风险、低价质损、售后风险、数据缺失

### 7.2 推荐等级

| 等级 | 含义 |
| --- | --- |
| P0 | 优先复核，可进入铺货候选 |
| P1 | 可观察，需要人工再判断 |
| P2 | 弱推荐，仅保留备选 |
| 不建议 | 风险高或关键字段不满足 |

## 8. 数据真实性与核验机制

### 8.1 核验状态

每个商品需要有整体核验状态：

- `unverified`：仅有列表页或搜索结果数据
- `verified`：关键字段已进入详情页核验
- `partial_verified`：部分关键字段已核验
- `verify_failed`：详情页打开失败、字段缺失或页面异常

### 8.2 字段级核验

关键字段需要记录字段级状态：

```json
{
  "field": "shipping_fee",
  "raw_value": "包邮",
  "normalized_value": 0,
  "source": "detail_page",
  "verified_at": "2026-06-13T00:00:00+08:00",
  "status": "verified"
}
```

### 8.3 核验失败处理

核验失败的商品不应直接丢弃，而是保留在导出表中，并标记：

- 核验失败原因
- 是否可人工复核
- 是否禁止进入铺货候选

## 9. 导出要求

### 9.1 Excel 导出

MVP 优先支持 Excel。每次采集任务导出一个工作簿：

- `选品结果`：核心商品清单
- `字段说明`：每列含义
- `标签配置`：本次任务使用的标签和规则
- `核验失败`：详情页核验失败或关键字段缺失商品

`选品结果` 工作表需要优先参照店雷达导出表结构组织字段，确保运营可以快速对照使用。系统字段可追加在店雷达字段之后，例如推荐分、风险提示、核验状态、人工复核状态、微信小店铺货建议等。

导出后进入人工复核阶段。人工复核阶段不回写原始采集事实，只补充复核结论、备注和铺货建议。

### 9.2 在线文档

P1 支持在线文档，优先考虑：

- 飞书表格
- 飞书多维表格

在线文档同步需要明确：

- 应用凭证
- 文档权限
- 表格字段映射
- 追加还是覆盖
- 同步失败重试
